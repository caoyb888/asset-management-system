## 二、测试策略

### 2.1 测试类型分工

| 测试类型 | 工具 | 说明 |
|----------|------|------|
| **Service 单元测试** | JUnit 5 + Mockito | Mock Mapper/Redis层，验证纯业务逻辑分支 |
| **Controller 集成测试** | MockMvc + 真实 MySQL + 嵌入式 Redis | 验证 HTTP 入参校验、事务完整性、数据库读写 |
| **数据权限专项测试** | JUnit 5 + Mockito | 独立测试5种权限范围的SQL条件注入逻辑 |
| **认证安全专项测试** | MockMvc + Redis | 端到端验证登录/Token/锁定/强制下线全流程 |

### 2.2 测试框架版本

```xml
<!-- pom.xml 依赖 -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-test</artifactId>
    <scope>test</scope>
    <!-- 包含: JUnit 5.x, Mockito 5.x, AssertJ, MockMvc -->
</dependency>
<dependency>
    <groupId>org.mockito</groupId>
    <artifactId>mockito-inline</artifactId>
    <scope>test</scope>
    <!-- 支持 static 方法 mock（SmCryptoUtil / SecurityUtil） -->
</dependency>
```

### 2.3 关键约定

- 集成测试使用 `@Transactional`，每个 `@Test` 方法结束后**自动回滚**，保证用例隔离
- 开发环境**永久基础数据**通过 `sql/test-data-system.sql` 脚本写入，不受事务回滚影响（数据 ID 固定在 91001~ 范围）
- Service 单元测试**不启动 Spring 容器**，执行速度 < 50ms / 用例
- 测试配置文件：`src/test/resources/application-test.yml`（禁用 Nacos、连接本地 MySQL、使用嵌入式 Redis）
- 认证相关测试需 Mock `SmCryptoUtil.sm2Decrypt()` / `SmCryptoUtil.sm3Verify()` 静态方法
- 数据权限测试需通过 `SecurityContextHolder` 注入模拟用户

## 六、测试数据维护规范

1. **永久数据**：`sql/test-data-system.sql` 使用固定 ID（91001~91199），脚本首行加 `SET FOREIGN_KEY_CHECKS=0;`，执行后恢复，幂等可重跑
2. **临时数据**：集成测试中通过 `@BeforeEach` 注入，随 `@Transactional` 自动回滚，不污染永久数据
3. **密码数据**：测试用户密码统一为 `Test@12345`，SM3 哈希后存储，SQL 注释中注明明文
4. **Redis 数据**：集成测试使用独立的 Redis DB（如 db=15），`@AfterEach` 清理测试 key
5. **禁止操作**：不得直接 `DELETE/TRUNCATE` 永久测试表，若需重置可重跑 SQL 脚本

## 九、风险与依赖

| 风险点 | 等级 | 应对措施 |
|--------|------|---------|
| SM2/SM3 静态方法难以 Mock | 中 | 使用 `mockito-inline` 支持 static mock |
| Redis 依赖影响测试稳定性 | 中 | 集成测试使用嵌入式 Redis 或独立 DB 隔离 |
| 数据权限 SQL 拼接复杂，难以从单元测试完全覆盖 | 中 | 单元测试覆盖逻辑分支 + 集成测试验证实际 SQL 执行 |
| 多角色权限并集逻辑边界多 | 中 | 按排列组合设计多角色场景测试数据 |
| 认证流程涉及 JWT 时效性 | 低 | Mock 时钟或使用较长有效期的测试 Token |
| 编码生成并发安全 | 低 | 单元测试验证逻辑，并发测试留在集成测试阶段 |
