## 一、缺陷清单

### F-001 · 菜单删除缺少角色绑定校验

| 属性 | 内容 |
|------|------|
| **优先级** | P1 |
| **涉及模块** | §4.7 菜单管理 |
| **跳过测试** | `MENU-U-08`（`SysMenuServiceTest.deleteMenu_boundToRole_throws`） |
| **相关文件** | `asset-system/src/main/java/com/asset/system/menu/service/impl/SysMenuServiceImpl.java` |

**问题描述**

`SysMenuServiceImpl.deleteMenu()` 当前只检查"是否存在子菜单"，未检查"菜单是否已被角色绑定"。若直接删除已授权给角色的菜单，会导致 `sys_role_menu` 中存在孤立的无效关联记录，前端路由也可能出现 403 问题。

**期望行为**

删除菜单前，查询 `sys_role_menu` 中是否存在该菜单的关联记录；若存在，抛出 `SysBizException`（提示"菜单已绑定角色，请先解除授权"）。

**实现建议**

```java
// SysMenuServiceImpl.deleteMenu() 补充校验
long boundCount = roleMenuMapper.countByMenuId(id);  // 新增 Mapper 方法
if (boundCount > 0) throw new SysBizException(SysErrorCode.MENU_BOUND_TO_ROLE);
```

需同步添加：
- `SysRoleMenuMapper.countByMenuId(Long menuId)` 方法
- `SysErrorCode.MENU_BOUND_TO_ROLE`（建议错误码 6071）

### F-002 · 机构新增缺少编码唯一性校验

| 属性 | 内容 |
|------|------|
| **优先级** | P2 |
| **涉及模块** | §4.3 机构管理 |
| **跳过测试** | `DEPT-U-03`（`SysDeptServiceTest.createDept_duplicateCode_throws`） |
| **相关文件** | `asset-system/src/main/java/com/asset/system/dept/service/impl/SysDeptServiceImpl.java` |

**问题描述**

`SysDeptServiceImpl.createDept()` 直接插入记录，未对 `deptCode` 做唯一性校验。若两个部门使用相同编码，数据库层面没有唯一索引约束（`sys_dept` 表 `dept_code` 字段无 UNIQUE KEY），会导致编码重复进库，影响后续按编码查询的逻辑（如商铺 Excel 导入时按部门编码 lookup）。

**期望行为**

新增部门时，若 `deptCode` 非空，校验同编码是否已存在；若存在，抛出 `SysBizException`（提示"部门编码已存在"）。

**实现建议**

```java
// SysDeptServiceImpl.createDept() 补充校验
if (StringUtils.hasText(dto.getDeptCode())) {
    Long codeCount = baseMapper.selectCount(new LambdaQueryWrapper<SysDept>()
            .eq(SysDept::getDeptCode, dto.getDeptCode()));
    if (codeCount > 0) throw new SysBizException(SysErrorCode.DEPT_CODE_EXISTS);
}
```

建议同步在 `sys_dept` 表的 `dept_code` 列添加唯一索引（允许 NULL，仅对非空值约束）：

```sql
ALTER TABLE sys_dept ADD UNIQUE KEY uk_dept_code (dept_code);
-- 或改为允许重复 NULL 的函数索引（MySQL 8.0+）
```

需同步添加：
- `SysErrorCode.DEPT_CODE_EXISTS`（建议错误码 6031）

### F-003 · 用户新增缺少手机号唯一性校验

| 属性 | 内容 |
|------|------|
| **优先级** | P2 |
| **涉及模块** | §4.2 用户管理 |
| **跳过测试** | `USER-U-03`（`SysUserServiceTest.createUser_duplicatePhone_throws`） |
| **相关文件** | `asset-system/src/main/java/com/asset/system/user/service/impl/SysUserServiceImpl.java` |

**问题描述**

`SysUserServiceImpl.createUser()` 仅校验用户名唯一性（`username`），未对手机号（`phone`）做唯一性校验。系统设计上手机号用于密码找回和短信通知，若允许重复录入，将导致消息发送对象不明确，以及未来手机号登录功能存在安全隐患。

**期望行为**

新增用户时，若 `phone` 非空，校验同手机号是否已被其他用户使用；若存在，抛出 `SysBizException`（提示"手机号已被占用"）。

**实现建议**

```java
// SysUserServiceImpl.createUser() 补充校验
if (StringUtils.hasText(dto.getPhone())) {
    Long phoneCount = baseMapper.selectCount(new LambdaQueryWrapper<SysUser>()
            .eq(SysUser::getPhone, dto.getPhone()));
    if (phoneCount > 0) throw new SysBizException(SysErrorCode.USER_PHONE_EXISTS);
}
```

需同步添加：
- `SysErrorCode.USER_PHONE_EXISTS`（建议错误码 6014）

> 注：`updateUser()` 同样需要排除自身 ID 后做手机号唯一性校验：`.ne(SysUser::getId, dto.getId())`
