### 任务 1.2 — 核销管理全栈（最高复杂度）

```java
// 提交核销申请（创建 write_off + detail，发起 OA 审批）
Long submitWriteOff(WriteOffCreateDTO dto)

// OA 审批通过回调（核心账务事务）
@Transactional
void approveCallback(String approvalId)
/*
  ① UPDATE fin_receivable: received_amount += 核销额, outstanding_amount -= 核销额
     → 更新 status（outstanding=0 → 2; 0 < outstanding < actual → 1）
  ② UPDATE fin_receipt: write_off_amount += 核销额
     → 更新 status（write_off=total → 2; 否则 → 1）
  ③ IF overpay_amount > 0:
       INSERT fin_prepay_transaction(trans_type=1)
       UPDATE fin_prepay_account.balance += overpay
  ④ UPDATE fin_write_off: status=1
*/

// 作废核销（逆向回滚事务）
@Transactional
void cancelWriteOff(Long writeOffId)

// 分页查询
IPage<FinWriteOff> pageQuery(WriteOffQueryDTO query)

// 查询可核销应收列表（status in 0,1 的应收记录）
List<FinReceivableVO> queryWritableReceivables(Long contractId)
```

## 七、开发风险与规避

| 风险 | 规避方案 |
|------|---------|
| 核销事务并发超扣 | 保证金 / 预收款余额更新强制 `SELECT FOR UPDATE` |
| FinReceivableServiceImpl 重写引入回归 | 重写前先为现有接口补充集成测试，重写后对比结果 |
| DB 重建导致测试数据丢失 | 先备份旧数据 `mysqldump`，再执行 v2 脚本 |
| 前端路由文件不存在白屏 | 阶段零 0.5 先建骨架占位，防止路由跳转崩溃 |
