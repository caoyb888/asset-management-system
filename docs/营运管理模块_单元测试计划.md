# 营运管理模块 单元测试计划

> **文档版本：** v1.0
> **创建日期：** 2026-03-06
> **模块：** asset-operation（端口 8003）
> **测试类型：** 引擎单元测试（直接 new）+ Service 单元测试（Mockito）+ Controller 集成测试（MockMvc + MySQL）
> **用例总量：** 98 个（引擎 22 + Service 36 + Controller IT 40）
> **重点关注：** 营运管理与招商管理（inv_*）、基础数据（biz_*）、财务（fin_*）的跨模块数据关联

## 一、测试策略

| 层次 | 框架 | 原则 |
|------|------|------|
| 计算引擎 | JUnit 5，直接 `new` 实例 | 纯函数，不依赖 Spring，验证数学精确性和策略分支 |
| Service 单元测试 | JUnit 5 + Mockito `@InjectMocks` | Mock 所有 Mapper/子 Service/Engine，只测业务逻辑和状态机 |
| Controller 集成测试 | `@SpringBootTest` + MockMvc | 启动完整 Spring 上下文，连接真实 MySQL，`@Transactional` 自动回滚 |

### 关键约束

- 所有集成测试 **永久测试数据 ID 固定在 92001 ~ 92099**，与招商测试数据（91xxx）和业务自增 ID 隔离
- 集成测试用 `@Transactional` 自动回滚，永久数据不被污染
- 排除自定义 JWT 安全配置，`@AutoConfigureMockMvc(addFilters = false)` 绕过过滤器
- Engine 类内部使用 `JdbcTemplate` 直连数据库查询招商/基础表，集成测试需真实 MySQL；Service 单元测试中 Mock Engine

## 二十四、风险与建议

### 高风险关注项

1. **解约写入招商/基础表（TSE-U-07/08, TRM-I-07）：** 解约生效时直接更新 `inv_lease_contract.status` 和 `biz_shop.shop_status`，这是唯一的跨模块写入操作。必须用 `@Transactional` 保证原子性，集成测试需验证回滚场景（如部分更新失败）。

2. **应收重算红冲逻辑（RRE-U-02）：** 已推送的应收需红冲（status=3）后新建（version递增），若红冲成功但新建失败会导致数据不一致。需测试事务回滚。

3. **浮动租金阶梯 JSON 解析（FRC-U-03）：** `formula_params` 的 JSON 格式依赖招商模块写入规范。需测试异常 JSON 格式的容错处理。

4. **品牌名称字段（已知 Bug）：** `biz_brand` 表品牌名称列为 `brand_name_cn` 而非 `brand_name`，所有 JOIN 查询需使用 `bb.brand_name_cn AS brand_name`。

5. **幂等键机制（LED-U-08）：** 推送应收的幂等键格式为 `receivable_{id}_{version}`，需验证 version 递增后幂等键变化（允许变更后重新推送）。

### 测试数据隔离策略

- **dev 基础数据：** 90001~90099（项目/商铺/商家/品牌）
- **inv 招商数据：** 91001~91099（合同/费项/账期）
- **opr 营运数据：** 92001~92099（台账/变更/解约/营收/客流/预警）
- 各模块 ID 段完全隔离，互不干扰
- 集成测试 `@Transactional` 自动回滚临时数据，永久数据不被污染
