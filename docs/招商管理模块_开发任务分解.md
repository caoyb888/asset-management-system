# 招商管理模块 · 开发步骤与任务分解

> **项目**：产城公司资产管理系统  
> **模块**：招商管理（inv_* 系列）  
> **技术栈**：Spring Boot 3.x · MyBatis-Plus · MySQL 8.0 · Vue 3 · Element Plus  
> **预估工期**：约 6 周（参考全项目 16 周总周期）

---

## 总体开发顺序原则

招商管理模块存在清晰的业务依赖链，必须按依赖方向从底层到上层推进：

```
配置层（cfg）→ 意向协议（inv_intention）→ 招商合同（inv_lease_contract）
                                          ↘ 开业审批（inv_opening_approval）
租决政策（inv_rent_policy）→ 租金分解（inv_rent_decomposition）
```

不能并行开发"意向协议"和"招商合同"，因为合同依赖意向的转化逻辑和数据结构。 建议先打通核心主干（意向→合同），再补充旁支（开业审批、租决、租金分解）。

## 第三阶段：租金计算引擎（第 2 周）

> 这是全模块技术难度最高的部分，必须在意向协议开发之前单独完成并通过单元测试，后续所有费用计算都依赖它。

### 任务 3.1 — 计算引擎核心设计

在 `investment/engine/` 包内实现：

```java
// 计算策略接口
public interface RentCalculateStrategy {
    BigDecimal calculate(RentCalculateContext context);
}

// 5种收费方式实现类
FixedRentStrategy           // 固定租金
FixedCommissionStrategy     // 固定提成
StepCommissionStrategy      // 阶梯提成（分阶段取各段金额之和）
HigherOfStrategy            // 两者取高（比较固定与提成）
OneTimeStrategy             // 一次性收费

// 策略路由器（通过 cfg_rent_scheme.strategy_bean_name 注入）
@Component
public class RentCalculateStrategyRouter {
    public RentCalculateStrategy route(String strategyBeanName) { ... }
}

// 计算上下文（封装所有入参）
@Data
public class RentCalculateContext {
    private BigDecimal unitPrice;
    private BigDecimal area;
    private BigDecimal commissionRate;
    private BigDecimal minCommissionAmount;  // 对应 min_commission_amount 字段
    private LocalDate stageStart;
    private LocalDate stageEnd;
    private JsonNode formulaParams;          // 对应 formula_params 字段
    ...
}
```

### 任务 3.2 — 日期计算与账期生成器

```java
// 账期生成器
public class BillingGenerator {
    List<BillingPeriod> generate(LocalDate start, LocalDate end, 
                                  PaymentCycle cycle, BillingMode mode);
}
```

- 使用 `java.time` API，处理跨月/跨年/闰年边界
- 首账期（预付/当期/后付）单独逻辑，生成后 `billing_type=1` 标记
- 所有金额计算用 `BigDecimal`，禁止 `double`

### 任务 3.3 — 计算引擎单元测试

> 必须在进入业务开发前完成，测试覆盖率 ≥ 80%。

测试矩阵（优先级 P0）：

| 场景 | 测试用例数 |
|------|-----------|
| 固定租金 × 多种面积 × 多种租期 | 6+ |
| 固定提成（含最低金额兜底） | 4+ |
| 阶梯提成（跨年阶段切分） | 6+ |
| 两者取高（固定 > 提成 / 提成 > 固定） | 4+ |
| 账期生成（月付/季付/年付 × 预付/后付） | 12+ |
| 闰年 2 月边界 | 2+ |

**验收标准**：全部单元测试通过，金额计算结果精确到分，无浮点误差。

## 第四阶段：意向协议（第 2 周末 ~ 第 3 周）

> 核心主干，优先级最高。

### 任务 4.1 — 意向协议后端 CRUD + 状态机

**接口**（对应技术报告 §3.1）：

- `GET /intentions`：分页列表，支持项目/状态/商家/楼栋/楼层/商铺/业态等多条件筛选（利用 `idx_intention_multi` 复合索引）
- `POST /intentions`：新增（草稿，status=0）
- `PUT /intentions/{id}`：修改（仅草稿/驳回状态可改）
- `DELETE /intentions/{id}`：逻辑删除（校验状态）
- `PUT /intentions/{id}/draft`：暂存
- `POST /intentions/{id}/submit-approval`：发起审批（status 0→1，调审批引擎）

状态机校验（Service 层拦截非法转换）：

```
草稿(0) → 审批中(1)：发起审批
审批中(1) → 审批通过(2)：审批回调
审批中(1) → 驳回(3)：审批回调
驳回(3) → 审批中(1)：重新发起
审批通过(2) → 已转合同(4)：转合同时修改
```

### 任务 4.2 — 商铺关联、费项配置、分铺计租

**接口**：
- `POST /intentions/{id}/shops`：关联商铺（写 `inv_intention_shop`）
- `POST /intentions/{id}/fees`：配置费项（写 `inv_intention_fee`，含 `formula_params`）
- `POST /intentions/{id}/fee-stages`：配置分铺计租阶段（写 `inv_intention_fee_stage`，含 `min_commission_amount`）
- `POST /intentions/{id}/generate-cost`：调用计算引擎，生成费用明细，更新 `total_amount`
- `POST /intentions/{id}/billing`：调用账期生成器，写 `inv_intention_billing`
- `GET /intentions/{id}/billing`：查询账期列表

关键业务规则：
- 生成费用时，将 `formula_params` 写入 `inv_intention_fee`，便于后续合同继承
- 账期生成前必须先完成费项配置和费用生成

### 任务 4.3 — 审批回调接口

- `POST /intentions/{id}/approval-callback`：接收审批引擎回调，更新 status + approval_id
- 审批通过时将商铺状态变更为"意向中"（联动 `biz_shop`，可通过事件或直接调用基础数据服务）

### 任务 4.4 — 意向协议前端（向导式表单）

按技术报告第四章页面清单实现：

**Step 1 - 计租方案选择**：调用计租方案选择器组件，选中后下一步

**Step 2 - 基础信息录入**：项目（必填）、意向名称、签约主体、品牌/商家（可选）

**Step 3 - 商务信息录入**：商铺选择（级联：楼栋→楼层→商铺）、交付日、装修期、开业日、合同起止日期、支付周期、账期模式

**Step 4 - 费项配置**：动态表格，按 `charge_type` 渲染不同列（固定显示单价/面积，提成显示比率/最低金额，阶梯显示阶段行），参考 §4.3 业务类型字典

**Step 5 - 分铺计租**：按已关联商铺生成 Tab，每个商铺下配置各阶段单价

**Step 6 - 费用生成**：点击"生成费用"调后端接口，展示费用明细汇总表（含 total_amount）

**Step 7 - 账期设置**：展示账期列表，首账期高亮，支持手动调整应收日期

**状态管理**：用 `useIntentionStore` 缓存全部步骤数据，步骤间可回退，修改上游步骤时弹确认（下游数据需重算）

**暂存**：每步"下一步"时自动调 `/draft` 接口持久化

**验收标准**：完整走通"创建意向协议→配置费项→生成费用→生成账期→发起审批"全流程。

## 第五阶段：招商合同（第 3 周末 ~ 第 4 周）

### 任务 5.1 — 意向转合同（核心难点）

这是整个模块技术复杂度最高的接口，需严格按以下步骤实现：

**接口**：`POST /contracts/from-intention/{intentionId}`

后端实现（事务 + 分布式锁）：

```java
@Transactional
public Long convertFromIntention(Long intentionId) {
    // 1. 校验意向状态（必须是"审批通过"）
    // 2. 校验商铺未被其他有效合同占用（查 inv_lease_contract_shop）
    // 3. 获取 Redis 分布式锁（key: shop:contract:{shopId}，TTL 30s）
    // 4. 再次校验（双重检查锁）
    // 5. 迁移数据：
    //    inv_intention → inv_lease_contract
    //    inv_intention_shop → inv_lease_contract_shop
    //    inv_intention_fee（含 formula_params）→ inv_lease_contract_fee
    //    inv_intention_fee_stage（含 min_commission_amount）→ inv_lease_contract_fee_stage
    //    inv_intention_billing → inv_lease_contract_billing（清空，重新生成）
    // 6. 更新意向状态为"已转合同"(status=4)
    // 7. 更新商铺状态为"已签约"
    // 8. 释放分布式锁
    // 9. 写合同版本快照至 inv_lease_contract_version
}
```

**前端**：
- 意向协议列表页"转合同"按钮（仅 status=2 审批通过状态可点）
- `/inv/contracts/from-intention`：展示意向数据预览 → 补录合同专属信息（合同类型等）→ 提交

### 任务 5.2 — 合同直接新增与管理

**接口**：
- `POST /contracts`：全新录入（流程同意向协议，合同专属字段：`contract_type`）
- `GET /contracts`：列表查询（含合同到期预警，利用 `idx_contract_end` 索引）
- `PUT /contracts/{id}/status`：变更合同状态（生效/终止）
- `POST /contracts/{id}/submit-approval`：发起审批

合同状态机：
```
草稿(0) → 审批中(1) → 生效(2) → 到期(3)（定时任务自动触发）
生效(2) → 终止(4)（人工操作）
```

### 任务 5.3 — 合同版本管理

每次合同修改（状态≠草稿时）：
- 旧记录 `is_current=0`
- 新记录 `version+1`，`is_current=1`
- 向 `inv_lease_contract_version` 写 `snapshot_data`

实现 `ContractVersionService.createSnapshot(Long contractId)` 工具方法。

### 任务 5.4 — 合同前端页面

- 合同列表页：状态标签（草稿/审批中/生效/到期/终止），到期前 30 天橙色预警
- 合同详情页：基本信息 + 费项 + 账期 Tab（复用意向协议子组件）
- 合同文本预览弹窗：富文本渲染 + 打印按钮

**验收标准**：意向转合同并发场景下只能成功一次（Redis 锁测试），合同版本快照可查询。

### 任务 9.1 — 前后端联调

按业务流程逐一联调：

1. 配置计租方案 → 创建意向协议（走完 7 步向导）→ 发起审批 → 审批通过
2. 意向转合同（含并发测试）→ 合同生效 → 合同到期预警
3. 开业审批 → 驳回 → 基于历史创建 → 重新提交 → 通过
4. 租决政策 → 审批通过 → 创建租金分解 → 导入/导出

### 任务 9.2 — 关键场景测试

| 测试场景 | 覆盖功能点 |
|---------|-----------|
