# 财务管理模块 单元测试计划

> **文档版本：** v1.0
> **创建日期：** 2026-03-09
> **模块：** asset-finance（端口 8004）
> **测试类型：** 公共基础设施单元测试（直接 new / Spring Context）+ Service 集成测试（H2 内存库）+ Controller 集成测试（MockMvc + H2）
> **用例总量：** 112 个（公共基础 8 + Service 52 + Controller IT 52）
> **重点关注：** 核销事务正确性、金额公式一致性、乐观锁并发控制、状态机流转、跨模块账户联动

## 一、测试策略

| 层次 | 框架 | 原则 |
|------|------|------|
| 公共基础设施 | JUnit 5，直接 `new` 或 `@SpringBootTest` | 验证单号生成唯一性、错误码体系、乐观锁重试 AOP、OA 适配器 Mock |
| Service 集成测试 | `@SpringBootTest(NONE)` + H2 内存库 | 真实 Service → Mapper → H2，验证业务逻辑、事务、金额公式 |
| Controller 集成测试 | `@SpringBootTest(MOCK)` + MockMvc + H2 | 完整 Spring 上下文，验证 HTTP 请求/响应、参数校验、端到端数据流 |

### 关键约束

- 所有测试使用 **H2 内存库**（MySQL 兼容模式），无需外部 MySQL 依赖
- 测试基类 `FinanceTestBase` 已配置：禁用 Nacos/Redis/Security，Mock OA 审批适配器
- 每个测试方法在独立事务中运行并 **自动回滚**，保证数据隔离
- H2 建表脚本 `schema-h2.sql` 包含全部 13 张 `fin_*` 表
- Service 测试通过 `@BeforeEach` 在每个用例前插入干净数据，不依赖永久测试数据
- Controller IT 使用 `@AutoConfigureMockMvc(addFilters = false)` 绕过 JWT 过滤器

### 与营运模块测试策略的差异

| 对比项 | 营运模块 (asset-operation) | 财务模块 (asset-finance) |
|--------|---------------------------|--------------------------|
| 数据库 | 真实 MySQL（跨模块 JdbcTemplate） | H2 内存库（自包含，无跨模块读写） |
| 跨模块依赖 | 读写 inv_*/biz_*/fin_* 表 | 无直接跨模块 SQL，通过接口调用 |
| 测试数据 | 永久数据 ID 92001~92099 | @BeforeEach 动态插入，自动回滚 |
| 引擎层 | 计算引擎单独测试（直接 new） | 无独立引擎类，计算逻辑内嵌 ServiceImpl |

### 跨模块数据依赖说明

```xml
    </configuration>

```

#### Controller IT 测试基类（需新建）

**`FinanceControllerTestBase.java`**

```java
@SpringBootTest(
    webEnvironment = SpringBootTest.WebEnvironment.MOCK,
    classes = FinanceApplication.class,
    properties = {
        "spring.cloud.nacos.discovery.enabled=false",
        "spring.cloud.nacos.config.enabled=false",
        "spring.cloud.nacos.config.import-check.enabled=false"
    }
)
@ActiveProfiles("test")
@AutoConfigureMockMvc(addFilters = false)
@Transactional
public abstract class FinanceControllerTestBase {

    @Autowired protected MockMvc mockMvc;
    @Autowired protected ObjectMapper objectMapper;
    @MockBean  protected OaApprovalAdapter oaApprovalAdapter;

    @BeforeEach
    void setupOaMock() {
        when(oaApprovalAdapter.submitApproval(anyString(), anyLong(), anyString()))
                .thenAnswer(inv -> "MOCK-APPROVAL-" + inv.getArgument(1));
    }
}
```

## 二十三、风险与建议

### 高风险关注项

1. **核销事务原子性（WOF-10, E2E-01~05）：** `approveCallback` 方法涉及更新 3 张表（receivable + receipt + prepay_account），任一步失败必须全部回滚。H2 内存库支持事务，但 `@Transactional` 测试中回滚行为与生产环境略有差异，建议补充 MySQL 环境回归测试。

2. **金额精度（全局）：** 所有金额字段使用 `BigDecimal` + `DECIMAL(14,2)`，测试中必须用 `isEqualByComparingTo` 而非 `isEqualTo` 比较（避免 scale 差异导致误判）。

3. **乐观锁并发（DEP-05, COM-05~06）：** H2 内存库的锁行为与 MySQL InnoDB 不完全一致，`SELECT FOR UPDATE` 在 H2 中可能不生效。保证金并发冲抵的真实压力测试需在 MySQL 环境执行。

4. **H2 与 MySQL 差异：** H2 的 `MODE=MySQL` 覆盖了大部分语法，但以下场景需注意：
   - `ON DUPLICATE KEY UPDATE` 在 H2 中为 `MERGE INTO` 语义
   - 全文索引 `FULLTEXT` 在 H2 中不支持
   - `UNSIGNED` 类型在 H2 中被忽略

5. **预收款自动创建（WOF-02, E2E-02）：** 核销超额时自动查找或创建 `fin_prepay_account`，需验证首次超额（账户不存在时自动创建）和二次超额（账户已存在时累加）两个路径。

### 测试数据策略

- **无永久测试数据依赖**：每个测试通过 `@BeforeEach` 插入干净数据，`@Transactional` 自动回滚
- **虚拟外键 ID**：使用不存在的 contractId/merchantId/projectId（10001~30099），不依赖其他模块真实数据
- **唯一编码**：通过 `System.nanoTime()` 后缀避免编码冲突
