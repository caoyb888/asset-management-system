# 流程管理轻量可视化配置器 — 开发任务计划

> **模块：** asset-workflow（端口 8010）+ asset-ui
> **目标：** 在流程定义管理页新增可视化审批链路设计器，支持通过拖拽/添加节点配置多级审批流程（如：申请人 → 部门审批 → 分管领导 → 主管领导），后端动态生成 BPMN XML，替代手工编辑 XML 的方式
> **编写日期：** 2026-03-23
> **优先级：** P1

---

## 一、现状分析

### 1.1 当前流程定义管理的局限

| 问题 | 说明 |
|------|------|
| 审批链路固化 | 8 种流程定义全部使用相同的三级金额网关（部门领导 → 副总裁 → 总经理），无法差异化配置 |
| 节点修改困难 | 必须手工粘贴 BPMN 2.0 XML，对业务人员不友好 |
| 审批人策略未启用 | `approver_strategy` / `approver_config` 字段在数据库中已存在，但未接入路由逻辑 |
| 新增业务类型门槛高 | 每新增一种审批场景需同时创建 BPMN XML 文件并部署，无法在线配置 |
| 条件分支不灵活 | 金额阈值（10万/50万）写死在 BPMN XML 中，无法调整 |

### 1.2 目标效果

用户在"流程定义管理"页面新增或编辑流程定义时，可通过**可视化节点卡片**完成如下配置：

```
┌──────────┐   ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│  发起申请  │──►│  部门主管审批  │──►│  分管领导审批  │──►│  主管领导审批  │
│ (固定节点) │   │  (部门自动解析)│   │  (指定角色)   │   │  (指定角色)   │
└──────────┘   └──────────────┘   └──────────────┘   └──────────────┘
                                        ▲
                                   [条件分支]
                               金额 ≥ 10万 才触发
```

保存后，系统自动将可视化配置转换为标准 BPMN 2.0 XML 并写入数据库，流程引擎直接加载使用。

## 二、总体方案设计

### 2.1 架构设计

```
前端（asset-ui）
  └── src/views/workflow/definition/index.vue
        └── FlowNodeDesigner.vue（新增可视化设计器组件）
              ├── StartNode（发起节点，固定）
              ├── ApproverNode（审批节点，可配置审批人策略）
              ├── ConditionNode（条件节点，可配置触发条件）
              └── EndNode（结束节点，固定）

后端（asset-workflow）
  └── WfDefinitionController
        └── POST /wf/definitions（增强：支持 nodeConfigs 字段）
  └── BpmnGeneratorService（新增：将 nodeConfigs 转换为 BPMN XML）
  └── WfNodeConfig（新增实体：持久化节点配置）

数据库（asset_db）
  └── wf_node_config（新增表：存储各流程定义的节点列表）
```

### 2.2 节点类型定义

| 节点类型 | 标识 | 配置项 | 说明 |
|----------|------|--------|------|
| 发起节点 | `START` | 无 | 固定节点，不可删除 |
| 审批节点 | `APPROVER` | 审批人策略、指定角色/用户、节点名称、超时时长 | 核心节点，支持多种审批人解析策略 |
| 条件节点 | `CONDITION` | 条件类型（金额/自定义）、比较运算符、阈值 | 前置条件，满足条件才进入后续节点 |
| 结束节点 | `END` | 无 | 固定节点，不可删除 |

### 2.3 审批人策略枚举（扩展现有字段）

| 策略值 | 中文名 | 说明 | 所需配置 |
|--------|--------|------|---------|
| `DEPT_LEADER` | 部门主管 | 发起人所在部门的直属主管（系统自动解析） | 无额外配置 |
| `ROLE` | 指定角色 | 具有指定角色的用户均可审批 | roleCode（如 `ROLE_VP`） |
| `SPECIFIC_USER` | 指定用户 | 固定指定某个用户审批 | userId |
| `INITIATOR_LEADER` | 发起人上级 | 发起人的直属上级用户（系统自动解析） | 无额外配置 |

## 三、数据库设计

### 3.1 新增表：wf_node_config（流程节点配置表）

```sql
CREATE TABLE wf_node_config (
    id              BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY COMMENT '主键',
    definition_id   BIGINT UNSIGNED NOT NULL COMMENT '关联 wf_process_definition.id',
    node_id         VARCHAR(64)     NOT NULL COMMENT '节点标识（BPMN element id）',
    node_type       VARCHAR(32)     NOT NULL COMMENT '节点类型：START/APPROVER/CONDITION/END',
    node_name       VARCHAR(128)    NOT NULL COMMENT '节点显示名称',
    node_order      INT             NOT NULL DEFAULT 0 COMMENT '节点排序（从0开始）',
    approver_strategy VARCHAR(32)             COMMENT '审批人策略：DEPT_LEADER/ROLE/SPECIFIC_USER/INITIATOR_LEADER',
    role_code       VARCHAR(64)              COMMENT '角色编码（策略=ROLE时使用）',
    user_id         BIGINT UNSIGNED          COMMENT '指定用户ID（策略=SPECIFIC_USER时使用）',
    condition_type  VARCHAR(32)              COMMENT '条件类型：AMOUNT/CUSTOM（节点类型=CONDITION时使用）',
    condition_op    VARCHAR(16)              COMMENT '运算符：GT/GTE/LT/LTE/EQ（≥ ≤ > < =）',
    condition_value DECIMAL(18,2)            COMMENT '条件阈值（金额条件时使用）',
    condition_expr  VARCHAR(512)             COMMENT '自定义 EL 表达式（自定义条件时使用）',
    timeout_hours   INT                      COMMENT '审批超时时长（小时），NULL=不限',
    remark          VARCHAR(256)             COMMENT '节点备注',
    created_by      BIGINT UNSIGNED          COMMENT '创建人ID',
    created_at      DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_by      BIGINT UNSIGNED          COMMENT '更新人ID',
    updated_at      DATETIME        DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    is_deleted      TINYINT         NOT NULL DEFAULT 0 COMMENT '逻辑删除：0正常 1删除',
    INDEX idx_definition (definition_id),
    INDEX idx_order (definition_id, node_order)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='流程节点配置表';
```

### 3.2 wf_process_definition 表补充说明

现有字段无需变更，新增字段关联关系：

### TASK-VF-02：新增 BpmnGeneratorService（核心）

**工作量：** 2 天

新建 `service/BpmnGeneratorService.java`，将节点列表转换为标准 BPMN 2.0 XML。

**输入：** `List<WfNodeConfig>` + `processKey` + `processName`

**输出：** BPMN 2.0 XML 字符串

**生成逻辑：**

```
节点列表 → 按 node_order 排序
  │
  ├─ 遍历每个节点
  │    ├─ START     → <startEvent id="start" name="发起申请"/>
  │    │
  │    ├─ APPROVER  → <userTask id="{node_id}" name="{node_name}">
  │    │               ├─ DEPT_LEADER  → flowable:assignee="${deptLeaderId}"
  │    │               ├─ ROLE         → flowable:candidateGroups="{roleCode}"
  │    │               ├─ SPECIFIC_USER→ flowable:assignee="{userId}"
  │    │               └─ INITIATOR_LEADER→ flowable:assignee="${initiatorLeaderId}"
  │    │
  │    ├─ CONDITION → <exclusiveGateway id="{node_id}"/>
  │    │               + <sequenceFlow conditionExpression="${amount >= {value}}"/>
  │    │               + <sequenceFlow conditionExpression="${amount < {value}}" targetRef="end"/>
  │    │
  │    └─ END       → <endEvent id="end" name="审批完成"/>
  │
  └─ 生成 <sequenceFlow> 连线（按 node_order 顺序连接）
```

**方法签名：**
```java
public interface BpmnGeneratorService {
    /**
     * 根据节点配置列表生成 BPMN XML
     */
    String generate(String processKey, String processName, List<WfNodeConfig> nodes);

    /**
     * 验证节点配置合法性（至少含1个审批节点、无悬空连线等）
     */
    void validate(List<WfNodeConfig> nodes);
}
```

**生成的 XML 模板（示例：3 节点流程）：**
```xml
<?xml version="1.0" encoding="UTF-8"?>
<definitions xmlns="http://www.omg.org/spec/BPMN/20100524/MODEL"
             xmlns:flowable="http://flowable.org/bpmn"
             targetNamespace="http://flowable.org/bpmn">
  <process id="OPR_CONTRACT_CHANGE" name="合同变更审批" isExecutable="true">
    <startEvent id="start" name="发起申请"/>
    <sequenceFlow id="flow_start" sourceRef="start" targetRef="node_1"/>

    <userTask id="node_1" name="部门主管审批" flowable:assignee="${deptLeaderId}"/>
    <sequenceFlow id="flow_1_to_gw" sourceRef="node_1" targetRef="gw_1"/>

    <exclusiveGateway id="gw_1" name="金额条件"/>
    <sequenceFlow id="flow_gw_skip" sourceRef="gw_1" targetRef="end" name="金额小于10万">
      <conditionExpression>${amount &lt; 100000}</conditionExpression>
    </sequenceFlow>
    <sequenceFlow id="flow_gw_next" sourceRef="gw_1" targetRef="node_2" name="金额大于等于10万">
      <conditionExpression>${amount &gt;= 100000}</conditionExpression>
    </sequenceFlow>

    <userTask id="node_2" name="分管领导审批" flowable:candidateGroups="ROLE_VP"/>
    <sequenceFlow id="flow_2_to_end" sourceRef="node_2" targetRef="end"/>

    <endEvent id="end" name="审批完成">
      <extensionElements>
        <flowable:executionListener event="end"
          class="com.asset.workflow.listener.ProcessCompleteListener"/>
      </extensionElements>
    </endEvent>
  </process>
</definitions>
```

### TASK-VF-04：Flowable 流程热部署

**工作量：** 0.5 天

修改 `WorkflowApprovalServiceImpl`，在保存流程定义后自动将新 BPMN XML 部署到 Flowable 引擎：

```java
// 每次保存 bpmn_xml 后调用
private void deployBpmnXml(String processKey, String bpmnXml) {
    Deployment deployment = repositoryService.createDeployment()
        .name(processKey)
        .addString(processKey + ".bpmn20.xml", bpmnXml)
        .deploy();
    log.info("流程定义部署成功: processKey={}, deploymentId={}", processKey, deployment.getId());
}
```

**注意事项：**
- 热部署不影响正在运行的流程实例（Flowable 版本化管理）
- 需在事务提交后执行部署（避免 DB 回滚但引擎已部署）
- 部署失败应回滚数据库操作并返回错误信息

## 八、关键设计决策

### 8.1 为什么不直接用 bpmn-js Modeler？

bpmn-js Modeler 功能完整但学习曲线陡，对业务人员不友好。轻量可视化配置器的核心优势：
- **业务语言**：节点类型用"部门主管审批"等业务语言描述，无 BPMN 专业术语
- **操作简单**：线性链路（不支持并行/子流程），适合资产管理场景
- **降级兼容**：保留 XML 源码编辑入口，技术人员仍可直接写 BPMN

### 8.2 条件节点的处理

条件节点（CONDITION）生成两条 sequenceFlow：
- **满足条件** → 流向下一个 APPROVER 节点
- **不满足条件** → 直接流向 END 节点（审批通过跳过后续节点）

若需要**不满足时驳回**（而非通过），可在条件节点增加 `onFalseAction: REJECT/SKIP` 配置项（V2 功能）。

### 8.3 向后兼容

- 现有 8 种流程定义的 BPMN XML 文件不受影响，仍可正常运行
- `wf_node_config` 表中无对应记录的流程定义，沿用 `bpmn_xml` 字段中的 XML
- 前端新 Dialog 对无 `nodeConfigs` 的旧流程默认显示"XML 源码"Tab

### 8.4 节点顺序约束

合法的节点序列规则（`BpmnGeneratorService.validate()` 负责校验）：
1. **必须以 START 开头、END 结尾**
2. **CONDITION 节点之后必须有 APPROVER 节点**（否则条件无意义）
3. **至少包含 1 个 APPROVER 节点**
4. **不能连续出现两个 CONDITION 节点**（暂不支持嵌套条件）
