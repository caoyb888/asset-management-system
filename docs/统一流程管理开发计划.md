# 统一流程管理开发计划

> **模块：** asset-workflow（端口 8010）
> **目标：** 将分散在招商、营运、财务三大模块中的 6+ 处审批逻辑，收敛为统一的流程管理中心
> **引擎：** Flowable 7.1.0（已在 parent pom 声明）
> **编写日期：** 2026-03-19

---

## 一、现状分析

### 1.1 当前审批分布

| 模块 | 业务场景 | 服务 | 适配器 | Mock 格式 | 回调 DTO |
|------|---------|------|--------|-----------|----------|
| 招商 | 意向协议审批 | InvIntentionServiceImpl | 内联 Mock | `MOCK-INV-{id}-{ts}` | 自定义 ApprovalCallbackDTO |
| 招商 | 开业审批 | InvOpeningApprovalServiceImpl | 未实现 | — | — |
| 招商 | 租金分解审批 | InvRentDecompositionServiceImpl | 内联 Mock | `MOCK-DECOMP-{id}-{ts}` | boolean approved |
| 营运 | 合同变更审批 | OprContractChangeServiceImpl | OaApprovalAdapter | `APPROVAL-{id}-{ts}` | ApprovalCallbackDTO |
| 营运 | 合同解约审批 | OprContractTerminationServiceImpl | OaApprovalAdapter | `TERM-APPROVAL-{id}-{ts}` | ApprovalCallbackDTO |
| 财务 | 核销审批 | FinWriteOffServiceImpl | OaApprovalAdapter | UUID | approvalId + boolean |
| 财务 | 减免审批 | FinReceivableServiceImpl | OaApprovalAdapter | UUID | approvalId + boolean |
| 财务 | 调整审批 | FinReceivableServiceImpl | OaApprovalAdapter | UUID | approvalId + boolean |

### 1.2 共同模式

所有审批场景均遵循相同的状态机：

```
DRAFT(0) ──[提交审批]──► APPROVING(1) ──[通过]──► APPROVED(2)
                                        ──[驳回]──► REJECTED(3) ──[重新编辑]──► DRAFT(0)
```

公共逻辑：
- **提交**：校验当前状态（仅 DRAFT/REJECTED 可提交）→ 生成 approvalId → 状态置为 APPROVING
- **回调**：幂等校验（非 APPROVING 忽略）→ 状态流转 → 触发业务引擎
- **并发控制**：`SELECT ... FOR UPDATE` + `@Version` 乐观锁

### 1.3 存在问题

| 问题 | 说明 |
|------|------|
| 代码重复 | `OaApprovalAdapter` 接口在 finance/operation 各复制一份，investment 未使用 |
| DTO 不统一 | `ApprovalCallbackDTO` 在 3 个模块各自定义，字段名/类型不一致 |
| 无审批记录 | 不记录审批人、审批意见、审批耗时，无法追溯 |
| 无统一入口 | 用户无法在一个页面看到"我的待办/已办"，需进入各业务模块分别查看 |
| Mock 硬编码 | 各模块各自生成 Mock approvalId，格式混乱，无法统一切换到真实 OA |
| workflow 空置 | `asset-workflow` 模块已引入 Flowable 7.1.0 依赖但完全空置 |

| 枚举值 | 说明 | 回调目标服务 | 回调后触发逻辑 |

/** Feign 接口 — 业务模块调用 workflow */
@FeignClient(name = "asset-workflow", path = "/wf")
public interface WorkflowFeignClient {

    /** 发起审批 */
    @PostMapping("/approvals/submit")
    R<String> submitApproval(@RequestBody ApprovalSubmitDTO dto);

    /** 查询流程状态 */
    @GetMapping("/approvals/{processInstanceId}/status")
    R<Integer> queryStatus(@PathVariable String processInstanceId);

    /** 撤回审批（仅发起人，且第一个节点未操作时可撤回） */
    @PostMapping("/approvals/{processInstanceId}/revoke")
    R<Void> revokeApproval(@PathVariable String processInstanceId);

    /** 查询审批记录（timeline） */
    @GetMapping("/approvals/{processInstanceId}/records")
    R<List<ApprovalRecordVO>> getRecords(@PathVariable String processInstanceId);

    /** 按业务单据查流程 */
    @GetMapping("/approvals/by-business")
    R<ProcessInstanceVO> getByBusiness(@RequestParam String businessType,
                                       @RequestParam Long businessId);
}

### 5.3 回调分发器

```java
/**
 * 审批完成后，根据 businessType 回调对应的业务模块。
 * 由 Flowable ProcessCompleteListener 触发。
 */
@Component
@RequiredArgsConstructor
public class ApprovalCallbackDispatcher {

    private final InvestmentCallbackClient investmentClient;
    private final OperationCallbackClient operationClient;
    private final FinanceCallbackClient financeClient;

    public void dispatch(ApprovalCallbackDTO dto) {
        String type = dto.getBusinessType();
        R<Void> result;

        if (type.startsWith("INV_")) {
            result = investmentClient.onApprovalCallback(dto);
        } else if (type.startsWith("OPR_")) {
            result = operationClient.onApprovalCallback(dto);
        } else if (type.startsWith("FIN_")) {
            result = financeClient.onApprovalCallback(dto);
        } else {
            throw new BizException("未知的业务类型: " + type);
        }

        if (result.getCode() != 200) {
            // 回调失败，记录日志 + 发送告警 + 支持重试
            log.error("审批回调失败: type={}, bizId={}, msg={}",
                type, dto.getBusinessId(), result.getMsg());
        }
    }
}
```

@Autowired
private ApprovalService approvalService;

public void submitApproval(Long changeId) {
    OprContractChange change = getById(changeId);
    if (change.getStatus() != 0 && change.getStatus() != 3)
        throw new BizException("仅草稿或驳回状态可提交审批");

    ApprovalSubmitDTO dto = new ApprovalSubmitDTO();

        .set(OprContractChange::getStatus, 1)
        .set(OprContractChange::getApprovalId, approvalId));

    private final OprContractTerminationService terminationService;

    @PostMapping("/approval-callback")

## 十、风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| Flowable 自动建表与现有表冲突 | 启动失败 | 配置 `flowable.database-schema-update: true`，Flowable 使用 ACT_ 前缀表，与 wf_ 业务表互不干扰 |
| 回调失败导致业务状态不一致 | 审批通过但业务未变更 | 回调失败时写入重试队列（Redis ZSET），定时任务兜底重试（最多 3 次） |
| 审批人配置不当（候选人为空） | 任务无法流转 | 每种业务类型配置默认审批角色，兜底分配给管理员 |
| Flowable 引擎性能 | 高并发下审批延迟 | 启用 Flowable async executor，历史数据定期归档 |
| 业务模块改造回归风险 | 现有功能异常 | Phase 3 先用 `approval.engine=mock` 回归，确认无误再切 flowable |
| Feign 调用超时 | 回调丢失 | Feign 超时设为 10s + 重试 2 次 + 失败告警 |
