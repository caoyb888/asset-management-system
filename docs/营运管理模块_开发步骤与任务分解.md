# 营运管理模块 · 开发步骤与任务分解

> **项目**：产城公司资产管理系统  
> **模块**：营运管理（Module 3）  
> **技术栈**：Spring Boot 3.x / MyBatis-Plus / MySQL 8.0 / Vue 3 / Element Plus  
> **参考文档**：技术分析报告 · 数据库设计（修订版）· CLAUDE.md  
> **建议周期**：8 周（含联调测试）

---

## 一、整体开发策略

营运管理模块围绕"合同台账"这一核心对象展开，所有子功能（变更、营收、客流、解约）都以台账为中枢进行关联。建议严格按照**数据层 → 服务层 → 接口层 → 前端层**的顺序逐子模块交付，优先保障 P0 功能，P1/P2 功能穿插在后期 Sprint 中完成。

整体遵循以下原则：
- **上游优先**：合同台账依赖招商合同模块数据，开发前需确认 `inv_lease_contract` 相关接口已具备联调条件
- **引擎前置**：应收重算引擎、浮动租金计算引擎、解约清算引擎是核心技术难点，单独作为基础组件在前期设计、中期开发，避免成为后期瓶颈
- **事务边界清晰**：多张表原子性更新（解约生效、变更生效）必须封装为独立事务性服务方法，不允许在 Controller 层处理
- **推送解耦**：财务系统推送全部走消息队列，不允许同步调用阻塞主流程

## 三、开发阶段划分

### 阶段一：合同台账（第 1–2 周）⭐ 核心主线

台账是整个营运模块的数据入口，其他功能均依赖台账的存在，必须首先完成。

#### 后端任务

**数据层**
- [ ] 创建 `OprContractLedger` 实体类及 Mapper，配置逻辑删除、乐观锁
- [ ] 创建 `OprReceivablePlan` 实体类及 Mapper，配置乐观锁
- [ ] 创建 `OprOneTimePayment` 实体类及 Mapper

**核心引擎（P0，必须先完成）**
- [ ] 实现 **应收计划生成器** `ReceivablePlanGenerator`：根据合同费项、计租周期、账期规则自动生成 `opr_receivable_plan` 明细，支持月付/季付/年付
- [ ] 实现 **财务推送服务** `ReceivablePushService`：封装 MQ 发送、幂等键生成（`receivable_{id}_{version}`）、推送状态回写；实现失败重试（指数退避）和手动重推入口

**业务服务层**
- [ ] `ContractLedgerService.generateFromContract(contractId)`：监听招商合同审批通过事件，自动创建台账记录（事件驱动）
- [ ] `ContractLedgerService.confirmDoubleSign(ledgerId)`：双签确认，更新 `double_sign_status` 和 `double_sign_date`
- [ ] `ContractLedgerService.generateReceivable(ledgerId)`：调用生成器创建应收计划，更新 `receivable_status`
- [ ] `ContractLedgerService.audit(ledgerId, pass/reject)`：审核台账，审核通过后触发推送
- [ ] `ContractLedgerService.addOneTimePayment(dto)`：一次性首款录入（单笔/多笔/历史账期三种模式），生成 `opr_receivable_plan` 记录（source_type=4）并回填 `receivable_id` 到 `opr_one_time_payment`
- [ ] 到期预警定时任务：XXL-Job 每日执行，扫描 `contract_end` 在 30/15/7 天内的台账，写入 `opr_alert_record`，调用消息服务发送

**接口层**（对应 API 设计 3.1 节）
- [ ] `GET /api/v1/operation/ledgers`：列表查询，支持 project_id / contract_type / status / contract_end / receivable_end / ledger_code 多条件分页
- [ ] `GET /api/v1/operation/ledgers/{id}`：详情（含费项、应收计划、商铺、双签状态）
- [ ] `PUT /api/v1/operation/ledgers/{id}/double-sign`
- [ ] `POST /api/v1/operation/ledgers/{id}/generate-receivable`
- [ ] `PUT /api/v1/operation/ledgers/{id}/audit`
- [ ] `POST /api/v1/operation/ledgers/{id}/push-receivable`（手动重推入口）
- [ ] `POST /api/v1/operation/ledgers/{id}/one-time-payment`
- [ ] `GET /api/v1/operation/ledgers/{id}/receivables`

#### 前端任务

- [ ] **台账列表页** `/opr/ledgers`：多条件筛选栏 + 分页表格，合同到期 ≤15 天标红预警色，应收到期列同样标色
- [ ] **台账详情页** `/opr/ledgers/:id`：Tab 布局（合同信息 / 应收计划 / 双签管理 / 审核记录），各 Tab 懒加载
- [ ] **应收计划面板**（嵌入组件）：明细表格 + 状态标签（待收/部分收/已收/已作废），推送状态标识，推送按钮（仅管理员可见）
- [ ] **一次性首款录入弹窗**：单笔/多笔/历史账期三 Tab 切换，表单校验，提交成功后刷新应收面板
- [ ] **合同到期预警页** `/opr/alerts/contract-expiry`：预警列表 + 剩余天数倒计时标签 + 快捷操作（跳转台账详情）

#### 验收标准

- 招商合同审批通过后，台账自动生成，无需手动操作
- 应收计划金额与合同费项一致，账期覆盖完整合同周期
- 推送财务系统幂等：同一应收重复推送，财务侧只处理一次
- 到期预警不重复发送（唯一键约束保障）

### 阶段二：合同变更（第 3 周）

变更功能依赖台账，且变更后的应收重算是本模块最高难度功能，需要充足的单元测试保障。

#### 后端任务

**数据层**
- [ ] 创建 `OprContractChange`、`OprContractChangeType`、`OprContractChangeDetail`、`OprContractChangeSnapshot` 实体及 Mapper

**核心引擎**
- [ ] 实现 **应收重算引擎** `ReceivableRecalculateEngine`：
  - 按变更类型分支处理：租金变更（重算未到期账期金额）、租期变更（新增/删除账期）、面积变更（全部费项重算）、单价变更（未到期账期）
  - 已推送应收采用"红冲+新增"模式：作废原 `opr_receivable_plan`（status=3），生成新记录（version 递增）
  - 变更前快照：JSON 序列化合同主表+费项+受影响应收，存入 `opr_contract_change_snapshot`

**业务服务层**
- [ ] `ContractChangeService.create(dto)`：新增变更单，`change_type` 写入 `opr_contract_change_type` 关联记录（替代逗号分隔）
- [ ] `ContractChangeService.previewImpact(changeId)`：预览变更影响（受影响应收笔数、金额差异），结果暂存至 `impact_summary`
- [ ] `ContractChangeService.submitApproval(changeId)`：提交 OA 审批，记录 `approval_id`
- [ ] `ContractChangeService.onApprovalCallback(approvalId, pass/reject)`：审批回调，通过后调用重算引擎；更新字段级变更明细（`opr_contract_change_detail`）
- [ ] 变更历史查询：按 `contract_id` 倒序，返回变更时间线数据

**接口层**（对应 API 设计 3.3 节）
- [ ] `GET /api/v1/operation/contract-changes`
- [ ] `GET /api/v1/operation/contract-changes/{id}`（含明细、对比、快照）
- [ ] `POST /api/v1/operation/contract-changes`
- [ ] `PUT /api/v1/operation/contract-changes/{id}`
- [ ] `POST /api/v1/operation/contract-changes/{id}/preview-impact`
- [ ] `POST /api/v1/operation/contract-changes/{id}/submit-approval`
- [ ] `GET /api/v1/operation/contract-changes/history/{contractId}`

#### 前端任务

- [ ] **变更列表页** `/opr/contract-changes`：状态筛选 + 变更类型标签（多值来自 `opr_contract_change_type`）
- [ ] **变更新增/编辑页** `/opr/contract-changes/form`：四步流程——合同选择 → 变更类型多选（Checkbox） → 动态表单（按类型渲染对应字段） → 影响预览确认
- [ ] **变更影响预览面板**（嵌入组件）：变更前后对比表（双列高亮显示差异），受影响应收差异汇总金额
- [ ] **变更历史时间线**（嵌入台账详情 Tab）：时间线组件展示每次变更，点击可展开变更明细和快照对比

#### 关键单元测试

- [ ] 租期延长后应收账期数量正确增加
- [ ] 面积变更后租金金额按比例重算准确
- [ ] 已推送应收的红冲+新增不产生重复推送
- [ ] 变更快照 JSON 可完整还原变更前状态

### 阶段三：营收填报 + 浮动租金（第 4 周）

营收填报相对独立，但浮动租金与应收计划深度关联，需在台账和变更完成后进行。

#### 后端任务

**数据层**
- [ ] 创建 `OprRevenueReport`、`OprFloatingRent`、`OprFloatingRentTier` 实体及 Mapper

**核心引擎**
- [ ] 实现 **浮动租金计算引擎** `FloatingRentCalculator`（策略模式）：
  - `FixedCommissionStrategy`：固定提成率计算
  - `TieredCommissionStrategy`：阶梯累进计算，每档明细写入 `opr_floating_rent_tier`（支持审计回溯）
  - `HigherOfStrategy`：取固定租金与提成两者较高值
  - 前置校验：月度填报完整性检查（是否覆盖该月每一天），不完整则拒绝计算
  - 计算结果：写入 `opr_floating_rent`，生成 `opr_receivable_plan`（source_type=3）

**业务服务层**
- [ ] `RevenueReportService.save(dto)`：按日录入营业额，唯一键重复时提示已存在
- [ ] `RevenueReportService.importExcel(file)`：批量导入，含格式校验（日期格式、金额范围）、异常值检测（偏差超历史均值3倍触发警告），返回成功行数和错误列表
- [ ] `RevenueReportService.exportExcel(query)`：导出营收报表
- [ ] `RevenueReportService.generateFloatingRent(contractId, month)`：触发浮动租金计算，月度数据不完整时返回明确错误
- [ ] `RevenueReportService.getMonthlyStatistics(query)`：按项目/楼栋/楼层/业态汇总

**接口层**（对应 API 设计 3.4 节）
- [ ] `GET /api/v1/operation/revenue-reports`
- [ ] `POST /api/v1/operation/revenue-reports`
- [ ] `PUT /api/v1/operation/revenue-reports/{id}`
- [ ] `POST /api/v1/operation/revenue-reports/import`（含文件上传）
- [ ] `GET /api/v1/operation/revenue-reports/export`
- [ ] `GET /api/v1/operation/revenue-reports/template`（模板下载）
- [ ] `POST /api/v1/operation/revenue-reports/generate-floating-rent`

#### 前端任务

- [ ] **营收填报列表页** `/opr/revenue-reports`：日期范围筛选 + 导入/导出按钮 + 状态过滤

### 阶段五：合同解约（第 5 周后半–第 6 周）

解约功能涉及多系统状态联动，是本模块第二高复杂度功能，需要完整的事务测试。

#### 后端任务

**数据层**
- [ ] 创建 `OprContractTermination`、`OprTerminationSettlement` 实体及 Mapper

**核心引擎**
- [ ] 实现 **解约清算引擎** `TerminationSettlementEngine`（策略模式）：
  - `NaturalTerminationStrategy`：到期解约，清算剩余未收费用
  - `EarlyTerminationStrategy`：提前解约，计算违约金（按日折算当期租金），清算未收费用
  - `RenewalTerminationStrategy`：重签解约，关联新合同 ID，清空原合同剩余应收
  - 通用：生成 `opr_termination_settlement` 明细（未收租金/违约金/保证金退还），正数为应收，负数为应退

**业务服务层**
- [ ] `ContractTerminationService.create(dto)`：新增解约单，初始状态草稿
- [ ] `ContractTerminationService.calculateSettlement(terminationId)`：调用清算引擎，生成明细，回填 `settlement_amount`
- [ ] `ContractTerminationService.submitApproval(terminationId)`：提交 OA 审批
- [ ] `ContractTerminationService.execute(terminationId)`：**事务性执行**，原子更新：
  - `opr_contract_ledger.status` → 2（已解约）
  - `inv_lease_contract.status` → 已终止（跨模块更新）
  - `biz_shop.shop_status` → 空置（跨模块更新）
  - `opr_receivable_plan` 未到期记录 status → 3（已作废）
  - MQ 通知财务系统作废对应应收
  - 取消未发送的到期预警（`opr_alert_record.sent_status` → 3）
- [ ] 应收到期预警定时任务（复用台账阶段基础设施，扫描 `opr_receivable_plan.due_date`）

**接口层**（对应 API 设计 3.6 节）
- [ ] `GET /api/v1/operation/terminations`
- [ ] `GET /api/v1/operation/terminations/{id}`（含清算明细、审批流程）
- [ ] `POST /api/v1/operation/terminations`
- [ ] `PUT /api/v1/operation/terminations/{id}`
- [ ] `POST /api/v1/operation/terminations/{id}/calculate-settlement`
- [ ] `POST /api/v1/operation/terminations/{id}/submit-approval`
- [ ] `POST /api/v1/operation/terminations/{id}/execute`

#### 前端任务

- [ ] **解约列表页** `/opr/terminations`：状态筛选 + 解约类型标签（到期/提前/重签不同颜色）
- [ ] **解约新增/编辑页** `/opr/terminations/form`：三步流程——合同选择 → 解约信息（类型/日期/原因） → 自动清算结果确认 → 提交审批
- [ ] **解约清算明细面板**（嵌入组件）：明细表格（未收租金/违约金/保证金退还/其他），正数标蓝（应收），负数标红（应退），底部汇总清算总额
- [ ] **解约详情页** `/opr/terminations/:id`：完整信息 + 清算明细 + 审批流程时间线

#### 关键事务测试

- [ ] 解约 execute 中任一更新失败，所有表均回滚
- [ ] MQ 发送失败不影响本地事务提交，异步补偿
- [ ] 重签解约后原合同商铺状态正确释放为空置

### 阶段七：联调与测试（第 7 周）

#### 集成联调

- [ ] **与招商合同模块联调**：触发合同审批通过事件 → 验证台账自动生成 → 应收计划金额校验
- [ ] **与财务系统联调**：应收推送 → 幂等键验证 → 财务侧状态回写 → 重试机制验证

## 五、关键风险与应对

| 风险点 | 等级 | 应对措施 |
|--------|------|----------|
| 招商合同模块接口未就绪，台账无法联调 | 高 | 第 0 周确认联调时间表；未就绪则先用 Mock 事件驱动开发，保留真实接口切换开关 |
| 财务系统推送协议变更 | 高 | 推送服务通过适配器模式隔离，接口变更只改适配层，不影响业务逻辑 |
| 应收重算引擎覆盖不全，导致线上金额错误 | 高 | 重算引擎 100% 单元测试覆盖，覆盖所有变更类型组合；上线前财务人员参与 UAT |
| 解约执行多表原子性失败 | 高 | 封装为 `@Transactional` 服务方法 + 人工干预入口 + 状态机防止二次执行 |
| 浮动租金阶梯计算逻辑分歧 | 中 | 开发前与业务方确认 3 种计费模式的边界案例（恰好在档位临界点时如何处理） |
| `opr_revenue_report` 唯一键软删除问题 | 中 | 应用层插入前校验 `is_deleted=0` 记录唯一性，已在数据库设计注释中标注 |

## 六、开发约定（对齐 CLAUDE.md）

- **代码注释**：全部使用中文，与 CLAUDE.md 规范一致
- **表命名**：营运模块统一前缀 `opr_`，与基础数据 `biz_`、招商 `inv_`、财务 `fin_` 区分
- **接口路径**：`/api/v1/operation/**`，统一返回 `{code, message, data}`
- **前端路由**：`/opr/**`，组件命名 PascalCase（如 `ContractLedgerList.vue`）
- **Git 分支**：`feature/opr-ledger`、`feature/opr-change`、`feature/opr-revenue`、`feature/opr-termination`，各子模块独立分支，完成后合入 `develop`
- **E2E 测试**：核心流程（台账生成 → 应收推送、变更审批 → 重算、解约执行 → 状态联动）使用 Playwright for Java 编写自动化测试用例
