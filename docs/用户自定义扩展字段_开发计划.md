# 用户自定义扩展字段 — 开发计划

> **项目**：产城公司资产管理系统
> **功能**：用户自定义扩展字段（JSON 方案）
> **技术栈**：Spring Boot 3 + MyBatis-Plus + Redis + Vue 3 + Element Plus
> **文档版本**：v1.0 · 2026-03-19
> **参考方案**：[用户自定义扩展字段方案.md](用户自定义扩展字段方案.md)

---

## 一、功能概览

允许管理员在系统管理界面为各业务模块自主配置扩展字段，无需开发介入。扩展字段以 JSON 形式存储在业务表的 `ext_fields` 列中，字段元数据统一由 `sys_ext_field_def` 表管理，前端通过通用组件动态渲染。

**支持模块（10个业务实体）：**

| module_code | 业务表 | 所属模块 |
|-------------|--------|---------|
| project | biz_project | 基础数据 |
| shop | biz_shop | 基础数据 |
| brand | biz_brand | 基础数据 |
| merchant | biz_merchant | 基础数据 |
| intention | inv_intention | 招商管理 |
| contract | inv_lease_contract | 招商管理 |
| ledger | opr_contract_ledger | 营运管理 |
| change | opr_contract_change | 营运管理 |
| receivable | fin_receivable | 财务管理 |
| receipt | fin_receipt | 财务管理 |

### TASK-EXT-01 | 数据库 DDL

**目标**：建立字段元数据表，为各业务表追加 `ext_fields` 列。

| 编号 | 任务 | 说明 |
|------|------|------|
| 01-01 | 创建 `sys_ext_field_def` 表 | 含 module_code、field_key、field_label、field_type、options_json、required、sort_order、show_in_list、show_in_form、default_val、placeholder、max_length、min_val、max_val、逻辑删除、审计字段；唯一索引 (module_code, field_key) |

| 01-04 | 营运管理模块业务表加列 | `ALTER TABLE opr_contract_ledger/opr_contract_change ADD COLUMN ext_fields JSON NULL` |
| 01-05 | 财务管理模块业务表加列 | `ALTER TABLE fin_receivable/fin_receipt ADD COLUMN ext_fields JSON NULL` |
| 01-06 | 编写迁移脚本 | 新建 `sql/ext_field_init.sql`，包含以上所有 DDL，可幂等重复执行 |

**交付物**：`sql/ext_field_init.sql`

### TASK-EXT-02 | 后端：字段定义管理（asset-system 模块）

**目标**：实现 `sys_ext_field_def` 的完整 CRUD，含 Redis 缓存。

#### 02-01 基础类

| 编号 | 任务 | 文件路径 | 说明 |
|------|------|----------|------|
| 02-01-A | Entity | `extfield/entity/SysExtFieldDef.java` | `@TableName("sys_ext_field_def")`，`optionsJson` 用 `JacksonTypeHandler` |
| 02-01-B | Mapper | `extfield/mapper/SysExtFieldDefMapper.java` | 继承 `BaseMapper<SysExtFieldDef>` |
| 02-01-C | Mapper XML | `resources/mapper/SysExtFieldDefMapper.xml` | 可选，复杂查询时补充 |
| 02-01-D | CreateDTO | `extfield/dto/ExtFieldCreateDTO.java` | 新增/修改共用；fieldKey 修改时禁止变更 |
| 02-01-E | VO | `extfield/dto/ExtFieldVO.java` | 对外返回，隐藏内部字段 |

#### 02-02 Service

| 编号 | 任务 | 说明 |
|------|------|------|
| 02-02-A | `SysExtFieldService` 接口 | 定义 list/getById/create/update/delete/updateSort |
| 02-02-B | `SysExtFieldServiceImpl` 实现 | 业务逻辑实现 |
| 02-02-C | list 方法 | 按 moduleCode 查询，按 sort_order 升序，命中缓存直接返回 |
| 02-02-D | create 方法 | 校验 (moduleCode, fieldKey) 唯一性；field_key 只允许英文+下划线（正则校验）；保存后删除 Redis 缓存 |
| 02-02-E | update 方法 | field_key 字段不允许修改（一旦有业务数据则不可变 key）；保存后删除 Redis 缓存 |
| 02-02-F | delete 方法 | 逻辑删除；清除 Redis 缓存；业务表中已有的 ext_fields 数据不做处理（前端不再渲染即可） |
| 02-02-G | updateSort 方法 | 接收 `List<{id, sortOrder}>`，批量更新 sort_order；清除 Redis 缓存 |
| 02-02-H | Redis 缓存 | Key：`sys:ext:field:{moduleCode}`，TTL：60分钟；新增/修改/删除/排序时 `evict` |

#### 02-03 Controller

| 编号 | 路径 | 方法 | 说明 |
|------|------|------|------|
| 02-03-A | `GET /sys/ext-fields` | list | 必传 moduleCode 参数；返回 `List<ExtFieldVO>` |
| 02-03-B | `GET /sys/ext-fields/{id}` | getById | 返回单条详情 |
| 02-03-C | `POST /sys/ext-fields` | create | `@Valid`；校验 fieldKey 格式 |
| 02-03-D | `PUT /sys/ext-fields/{id}` | update | `@Valid`；fieldKey 不可修改 |
| 02-03-E | `DELETE /sys/ext-fields/{id}` | delete | 逻辑删除 |
| 02-03-F | `PUT /sys/ext-fields/sort` | updateSort | 接收 `List<{id,sortOrder}}` |

## 五、关键技术约定

### 后端

1. **JacksonTypeHandler**：`SysExtFieldDef.optionsJson` 和所有业务 Entity 的 `extFields` 均使用此 TypeHandler，需在 `@TableName(autoResultMap = true)` 配合使用
2. **fieldKey 命名规范**：只允许小写英文字母、数字、下划线，不得以数字开头；正则：`^[a-z][a-z0-9_]{0,62}$`
3. **单模块最大字段数**：≤ 20，超出时 create 接口返回 `SysBizException`
4. **缓存 Key 格式**：`sys:ext:field:{moduleCode}`（与方案文档保持一致）
5. **后端二次校验**：保存业务数据时，根据 Redis 中的字段元数据对 `extFields` 做必填校验，防止绕过前端直接调用 API 时漏传必填字段

### 前端

1. **内存缓存**：`useExtFields` 内部 `Map<string, ExtFieldDef[]>` 缓存，同一页面多次调用同一 moduleCode 不重复请求
2. **组件注册**：`ExtFieldRenderer` 和 `ExtFieldColumns` 在 `main.ts` 中全局注册，无需每个页面单独引入
3. **v-model 绑定**：`ExtFieldRenderer` 使用 `v-model="form.extFields"`，父表单 el-form `:model` 须包含 `extFields` 字段才能触发内部 rules 校验
4. **空值兼容**：`form.extFields` 初始为 `{}`（非 null），避免 `v-model` 绑定时报错

## 七、风险项与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 业务表 ALTER TABLE 在生产环境锁表 | 低（JSON 列仅新增） | MySQL 8 的 `ADD COLUMN JSON NULL` 为 Online DDL，影响极小；生产执行在低峰期 |
| Entity 加 `extFields` 后已有接口响应体变化 | 低 | 新字段默认 null，前端响应解包层已存在，不影响现有功能 |
| 扩展字段必填校验被绕过 | 中 | 后端 Service 层根据 Redis 元数据做二次校验 |
| 字段定义被误删导致历史数据"不可读" | 低 | 软删除不影响 DB 数据；可提供"恢复字段定义"操作 |
