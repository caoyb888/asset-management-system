## 一、整体开发策略

### 1.1 核心原则

财务模块是整个资产管理系统中**数据一致性要求最高、业务规则最复杂**的模块。其核心难点在于"钱账对齐"——资金源（收款单/保证金账户/预收款账户）与欠款项（应收台账）的动态匹配，以及多表状态的原子同步。

基于此，制定以下开发策略：

**策略一：底层优先，由内向外**
先完成数据库、公共服务、审批流对接等基础设施，再开发业务逻辑，最后实现前端界面。避免因底层不稳定导致上层反复返工。

**策略二：核心链路优先，周边功能后置**
优先完成"收款→核销→应收→账户"的主链路，保障资金流转的完整闭环，减免/调整/凭证/报表等周边功能在主链路稳定后跟进。

**策略三：测试与开发并行，验收前置**
核心账务逻辑（事务、锁、状态机）必须在开发完成时同步编写单元测试，不推迟到最后统一测试，确保每个里程碑交付的代码质量可控。

**策略四：前后端分离，接口契约先行**
后端先输出接口文档（Knife4j），前端基于 Mock 数据并行开发，在联调阶段合并，缩短等待时间。

### 1.2 模块分层架构

```
┌─────────────────────────────────────────────────────────┐
│                    前端展示层（Vue3）                      │
│   收款单页面 · 核销操作 · 应收台账 · 账户余额 · 凭证管理    │
├─────────────────────────────────────────────────────────┤
│                    业务服务层（Spring Boot）               │
│   收款Service · 核销Service · 应收Service · 账户Service   │
├────────────────┬────────────────┬────────────────────────┤
│   事务管理层    │   审批流集成层   │      定时任务层          │
│  @Transactional│  OA回调/补偿    │  欠费预警/对账巡检        │
├─────────────────────────────────────────────────────────┤
│                    数据访问层（MyBatis-Plus）              │
│  乐观锁 · 行级锁(SELECT FOR UPDATE) · 逻辑删除            │
├─────────────────────────────────────────────────────────┤
│              MySQL 8.0  /  Redis  /  XXL-Job             │
└─────────────────────────────────────────────────────────┘
```

### 1.3 关键技术约束

| 约束项 | 要求 | 来源 |
|--------|------|------|
| 事务原子性 | 多表更新必须封装在同一 `@Transactional`，任一失败全部回滚 | 流程图-架构师核对意见 |
| 行级锁 | 保证金/预收款余额更新前执行 `SELECT FOR UPDATE` | 状态机手册-账户余额规则 |
| 乐观锁 | 核心收款/核销/应收/账户表均含 `version` 字段，并发冲突重试最多3次 | 数据库设计 |
| 幂等性 | 审批回调接口、凭证上传接口必须幂等（`approval_id` / `upload_status` 防重） | 状态机手册-QA要求 |
| 金额精度 | 所有金额字段 `DECIMAL(14,2)`，禁止使用浮点数计算 | 数据库设计 |
| original_amount | 应收台账原始金额字段**不可修改**，减免/调整只更新 `deduction_amount` / `adjust_amount` | 数据库设计 |

### 1.4 整体时间规划

```
第1周（D01-D05）  ── 前置准备
第2-3周（D06-D15）── 开发阶段一：核心主链路（收款 + 核销 + 应收台账）
第4-5周（D16-D25）── 开发阶段二：账户管理 + 凭证管理 + 周边功能

```

### 2.3 OA 审批流对接接口

**负责**：后端  **工期**：D02-D04  **工作量**：2 MD  **优先级**：P0

财务模块多个业务单据需要 OA 审批驱动状态流转（核销单、减免单、保证金冲抵等），审批流对接是整个模块的核心依赖。

**交付物**：
- 发起审批接口（向 OA 系统推送审批任务，返回 `approval_id`）
- 审批回调接口（OA 审批通过/驳回后回调，必须幂等：同一 `approval_id` 重复回调只处理一次）
- Mock 审批服务（供本地开发/测试使用，模拟审批通过/驳回两种场景）
- `approval_id` 存储与状态绑定设计

**回调处理逻辑**：

```
收到回调(approval_id, result)
  → 查询单据当前状态（必须为"待审核"才处理，否则直接返回成功，幂等）
  → result == 通过 → 触发账务事务（见各模块详细说明）
  → result == 驳回 → 解除冻结，单据状态置为"驳回"
```

**验收标准**：Mock 服务可正常触发回调；幂等测试（同一 `approval_id` 回调2次，数据库仅更新一次）通过。

## 三、开发阶段一：核心主链路（第2-3周 D06-D15）

> **目标**：打通"收款→核销→应收台账"的完整资金流转闭环，这是财务模块最核心、风险最高的部分。阶段一结束时，财务人员可以完成一笔完整的收款核销操作。

### 3.1 收款管理

**负责**：后端 + 前端  **工期**：D06-D10  **工作量**：后端 4.5 MD + 前端 3 MD  **优先级**：P0

#### 3.1.1 收款单后端

**功能列表**：

| 接口 | 方法 | 路径 | 说明 |
|------|------|------|------|
| 新增收款单 | POST | `/api/finance/receipt` | 含费项拆分明细，拆分合计必须等于总金额 |
| 编辑收款单 | PUT | `/api/finance/receipt/{id}` | 仅"待核销"状态可编辑 |
| 查看收款单详情 | GET | `/api/finance/receipt/{id}` | 含明细列表 |
| 分页查询收款单 | GET | `/api/finance/receipt/page` | 支持按合同、商家、日期、状态筛选 |
| 作废收款单 | PUT | `/api/finance/receipt/{id}/cancel` | 仅"待核销（status=0）"可作废 |
| 未名款归名 | PUT | `/api/finance/receipt/{id}/bind` | 将 `is_unnamed=1` 的收款单绑定合同 |

**关键业务规则**：
- `fin_receipt_detail` 的 `amount` 之和必须等于 `fin_receipt.total_amount`，后端强制校验
- 收款单状态机：`0(待核销) → 1(部分核销) → 2(已全部核销)`；仅 `status=0` 可作废为 `status=3`
- 未名款项（`is_unnamed=1`）归名后自动关联合同信息

**状态机单测要求**（与开发同步完成）：
- 合法路径：0→1→2 全流程
- 非法路径：已核销单据发起作废 → 期望抛出 `BizException(RECEIPT_ALREADY_WRITTEN_OFF)`
- 边界值：拆分明细合计不等于总金额 → 期望返回 400

#### 3.1.2 收款单前端

**页面清单**：

- **收款单列表页**（`/finance/receipt`）：多条件筛选栏（合同号/商家/日期范围/核销状态）、数据表格（含状态标签、已核销金额进度条）、新增/查看/作废操作列
- **新增/编辑抽屉**：基本信息表单 + 费项拆分明细动态表格（可增减行，实时合计校验提示）
- **作废确认弹窗**：二次确认 + 作废原因输入

**验收标准**：费项拆分合计实时联动，超出/不足时表单禁止提交并给出明确提示。

### 3.2 核销管理（核心，最高复杂度）

**负责**：高级后端 + 前端  **工期**：D09-D15  **工作量**：后端 7 MD + 前端 4 MD  **优先级**：P0

#### 3.2.1 核销主逻辑后端

这是整个财务模块**最核心、最复杂**的部分，必须由有 MySQL 事务与锁经验的工程师负责。

**核销类型（write_off_type）**：

| 类型值 | 类型名称 | 资金来源 | 特殊处理 |
|--------|----------|----------|----------|
| 1 | 收款核销 | `fin_receipt` | 超额部分自动转预存款 |
| 2 | 保证金冲抵 | `fin_deposit_account` | 行级锁 + 冲抵流水 |
| 3 | 预收款核销 | `fin_prepay_account` | 行级锁 + 抵冲流水 |
| 4 | 负数核销 | 冲红/退款场景 | `write_off_amount` 存负数，方向严格校验 |

**核销主流程（账务更新事务）**：

```
提交核销申请
  → 金额校验：∑write_off_detail.amount ≤ receipt.total_amount - receipt.write_off_amount
  → 创建 fin_write_off + fin_write_off_detail（status=0:待审核）
  → 发起 OA 审批（存储 approval_id）

OA 审批通过回调
  → @Transactional 开启事务
    ├─ UPDATE fin_receivable: received_amount += 核销金额, outstanding_amount -= 核销金额
    │   └─ 更新 status（outstanding=0 → 2:已收清；0<outstanding<actual → 1:部分收款）
    ├─ UPDATE fin_receipt: write_off_amount += 核销金额
    │   └─ 更新 status（write_off=total → 2:全部核销；否则 → 1:部分核销）
    ├─ IF 超额: INSERT fin_prepay_transaction(type=1) + UPDATE fin_prepay_account.balance
    └─ UPDATE fin_write_off: status=1:审核通过
  → 事务提交（任一失败全部回滚）
```

**负数核销特别处理**：
- `write_off_amount` 存储为**负数**
- Service 层必须校验：负数核销只能冲销已存在的正数核销记录，防止余额被错误累加
- 对应的 `fin_receivable.received_amount` 回减，`outstanding_amount` 回增

**接口清单**：

| 接口 | 说明 |
|------|------|
| POST `/api/finance/write-off` | 提交核销申请（含明细） |
| GET `/api/finance/write-off/page` | 分页查询核销单 |
| GET `/api/finance/write-off/{id}` | 查看核销单详情 |
| PUT `/api/finance/write-off/{id}/cancel` | 作废核销单（逆向回滚） |
| POST `/api/finance/write-off/callback` | OA 审批回调（幂等） |
| GET `/api/finance/write-off/receivable-list` | 查询可核销的应收列表 |

**逆向回滚（作废核销单）**：
```
作废核销单
  → @Transactional
    ├─ 逆向还原 fin_receivable.received_amount（减回核销金额）
    ├─ 逆向还原 fin_receipt.write_off_amount（减回核销金额）
    ├─ IF 曾产生超额预存: 逆向扣减 fin_prepay_account.balance
    └─ UPDATE fin_write_off: status=作废
```

**单元测试要求（必须与开发同步完成）**：
- 收款核销完整事务测试（验证3张表同时更新正确）
- 超额转预存测试（收款1000，核销800，验证 prepay_account.balance 增加200）
- 负数核销方向校验（正数应收用负数核销被拦截）
- 事务回滚测试（Mock 流水 INSERT 失败，验证余额未变更）
- 并发核销测试（同一收款单并发2个核销请求，乐观锁保证结果正确）

#### 3.2.2 核销前端

- **核销单列表页**：按核销类型/审批状态/合同筛选，展示审批进度标签
- **核销操作弹窗**：加载可核销的应收明细列表，按行录入核销金额，实时计算剩余可核销额，超额提示并显示预计转预存金额
- **审批进度展示**：待审核/审核通过/已驳回三态，通过后展示账务生效时间

### 3.3 应收台账管理

**负责**：后端 + 前端  **工期**：D08-D15  **工作量**：后端 5.5 MD + 前端 3 MD  **优先级**：P0

#### 3.3.1 应收台账后端

应收台账由营运模块的合同台账自动生成，财务模块主要负责查询、减免、调整。

**关键字段计算逻辑**（不可违反）：
```
actual_amount = original_amount + adjust_amount - deduction_amount
outstanding_amount = actual_amount - received_amount
```

**接口清单**：

| 接口 | 说明 |
|------|------|
| GET `/api/finance/receivable/page` | 多条件分页查询（合同/项目/商家/费项/账期/状态） |
| GET `/api/finance/receivable/summary` | 按合同/项目汇总（应收合计/已收/欠费） |
| GET `/api/finance/receivable/{id}` | 查看应收详情 |
| POST `/api/finance/receivable/deduction` | 新增减免申请 |
| PUT `/api/finance/receivable/deduction/{id}/callback` | 减免审批回调（通过后更新 `deduction_amount`，不改 `original_amount`） |
| POST `/api/finance/receivable/adjustment` | 新增调整申请（增加/减少） |
| PUT `/api/finance/receivable/adjustment/{id}/callback` | 调整审批回调（通过后更新 `adjust_amount`） |
| GET `/api/finance/receivable/export` | 应收明细 Excel 导出 |
| POST `/api/finance/receivable/print` | 账单打印（生成 PDF） |

**减免/调整业务规则**：
- 减免审批通过后：`deduction_amount += 减免金额`，`actual_amount` 重新计算，`original_amount` **严禁修改**
- 调整审批通过后：`adjust_type=1（增加）` 则 `adjust_amount += 调整金额`；`adjust_type=2（减少）` 则 `adjust_amount -= 调整金额`
- 减免后应收状态变为 `status=3:已减免`（当 `outstanding_amount > 0` 时也可继续核销）

**欠费预警定时任务（XXL-Job）**：
- 每日 08:00 扫描 `due_date < 今日` 且 `status in (0,1)` 的应收记录
- 发送站内消息通知财务人员（含逾期天数、金额、合同信息）

#### 3.3.2 应收台账前端

- **应收台账列表页**：顶部汇总卡片（本期应收/已收/欠费/减免），多条件筛选，状态徽标，操作列含"核销"快捷入口
- **减免申请弹窗**：选择减免原因，填写金额（不超过欠费金额），提交后展示审批状态
- **调整申请弹窗**：选择调整类型（增/减），填写原因和金额，提交审批

**接口清单**：

| GET `/api/finance/deposit/transaction/page` | 分页查询流水明细 |

| POST `/api/finance/deposit/refund` | 发起退款申请 |
| POST `/api/finance/deposit/forfeit` | 发起罚没申请 |
| PUT `/api/finance/deposit/callback` | 审批回调（处理冲抵/退款/罚没生效） |

**冲抵核心事务（审批通过后）**：
```java
@Transactional
public void processOffsetApprove(Long writeOffId) {
    // 1. 行级锁锁定账户（防并发超扣）
    FinDepositAccount account = depositAccountMapper
        .selectForUpdate(writeOffId);  // SELECT ... FOR UPDATE
    
    // 2. 校验余额
    if (account.getBalance().compareTo(offsetAmount) < 0) {
        throw new BizException(DEPOSIT_BALANCE_INSUFFICIENT);
    }
    
    // 3. 插入冲抵流水（trans_type=2，记录 balance_after）
    insertTransaction(account, OFFSET, offsetAmount);
    
    // 4. 更新账户余额（total_offset += offsetAmount）
    updateAccountBalance(account, offsetAmount);
    
    // 5. 更新应收台账状态
    updateReceivableStatus(writeOffId);
    
    // 以上操作在同一事务，任一失败全部回滚
}
```

**对账定时任务（XXL-Job）**：每日凌晨1:00，遍历所有账户验证余额恒等式，不一致时写入告警日志并推送通知。

**上传幂等性保证**（关键）：
```
上传前检查：upload_status == 0（未上传）
上传成功后：@Transactional

  → 若 affected_rows == 0，说明已被其他线程上传，直接返回成功（幂等）
```

**借贷平衡校验**：提交审核前后端强制校验 `total_debit == total_credit`，不平衡返回 `VOUCHER_UNBALANCED` 错误码。
