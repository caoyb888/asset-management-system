## 一、架构概览

```
┌─────────────────┐     ApprovalService.submit()      ┌───────────────────┐
│   业务模块       │ ─────────────────────────────────→ │  asset-api-workflow │
│ (inv/opr/fin)   │                                    │  (公共接口模块)     │
│                 │ ←─────────────────────────────────  │                   │
│                 │   ApprovalCallbackDTO (回调)         └───────┬───────────┘
└─────────────────┘                                            │
                                                               │ 两种实现
                                              ┌────────────────┼────────────────┐
                                              ▼                                 ▼
                                   ┌──────────────────┐              ┌──────────────────┐
                                   │ MockApprovalService│              │FlowableApproval  │
                                   │ (默认，自动通过)    │              │Service (真实引擎) │
                                   └──────────────────┘              └──────────────────┘
                                                                              │
                                                                              ▼
                                                                   ┌──────────────────┐
                                                                   │  asset-workflow   │
                                                                   │  (Flowable引擎)   │
                                                                   │  端口 8010        │
                                                                   └──────────────────┘
```

**核心设计原则：**

1. **业务模块只依赖 `asset-api-workflow`（接口模块），不依赖 `asset-workflow`（引擎模块）**
2. **Mock/Flowable 通过配置切换**，业务代码零修改
3. **回调统一使用 `ApprovalCallbackDTO`**，result=2 通过、result=3 驳回

## 二、后端接入（4 步）

### Step 1：添加 Maven 依赖

在业务模块的 `pom.xml` 中添加：

```xml
<dependency>
    <groupId>com.asset</groupId>
    <artifactId>asset-api-workflow</artifactId>
</dependency>
```

> 该模块提供 `ApprovalService` 接口、`ApprovalSubmitDTO`、`ApprovalCallbackDTO` 等类，无需引入 Flowable。

### Step 2：配置审批引擎

在业务模块的 `application-dev.yml` 中添加：

```yaml
# 审批引擎切换（mock=自动通过, flowable=走workflow服务）
approval:
  engine: mock
```

| 配置值 | 行为 | 适用场景 |
|--------|------|---------|
| `mock`（默认） | 直接返回 Mock ID，不调用外部服务 | 本地开发、单元测试、无需审批流程时 |
| `flowable` | 通过 Feign 调用 asset-workflow 启动真实流程 | 联调测试、生产环境 |

> **未配置时默认走 Mock 模式**，无需启动 asset-workflow 服务即可正常开发。

### Step 3：Service 层提交审批

在 Service 中注入 `ApprovalService` 并调用 `submit()` 方法：

```java
import com.asset.api.workflow.ApprovalService;
import com.asset.api.workflow.dto.ApprovalSubmitDTO;

@Service
@RequiredArgsConstructor
public class YourBusinessServiceImpl implements YourBusinessService {

    private final ApprovalService approvalService;

    @Override
    @Transactional(rollbackFor = Exception.class)
    public void submitApproval(Long id) {
        // 1. 查询业务单据并校验状态
        YourEntity entity = getById(id);
        if (entity.getStatus() != Status.DRAFT.getCode()
                && entity.getStatus() != Status.REJECTED.getCode()) {
            throw new BizException("当前状态不允许发起审批");
        }

        // 2. 构建审批提交 DTO
        ApprovalSubmitDTO dto = new ApprovalSubmitDTO();
        dto.setBusinessType("YOUR_BUSINESS_TYPE");   // 必填：业务类型枚举值
        dto.setBusinessId(id);                        // 必填：业务单据 ID
        dto.setTitle("XXX审批-" + entity.getCode());  // 必填：审批标题
        dto.setProjectId(entity.getProjectId());      // 可选：项目 ID（数据权限）
        dto.setPriority(0);                           // 可选：0普通 1紧急 2加急

        // 可选：扩展变量（用于多级审批金额网关判断）
        // dto.setVariables(Map.of("amount", entity.getAmount()));

        // 3. 提交审批，获取流程实例 ID
        String approvalId = approvalService.submit(dto);

        // 4. 更新业务单据状态为"审批中"，保存审批 ID
        update(new LambdaUpdateWrapper<YourEntity>()
                .eq(YourEntity::getId, id)
                .set(YourEntity::getStatus, Status.APPROVING.getCode())
                .set(YourEntity::getApprovalId, approvalId));
    }
}
```

**ApprovalSubmitDTO 字段说明：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| businessType | String | **是** | 业务类型，如 `INV_INTENTION`、`OPR_CONTRACT_CHANGE` |
| businessId | Long | **是** | 业务单据主键 ID |
| title | String | **是** | 审批标题，建议格式：`{业务名}-{单据编号}` |
| initiatorId | Long | 否 | 发起人 ID（可自动从 SecurityContext 获取） |
| initiatorName | String | 否 | 发起人姓名 |
| projectId | Long | 否 | 所属项目 ID（数据权限过滤） |
| priority | Integer | 否 | 优先级：0=普通 1=紧急 2=加急 |
| variables | Map | 否 | 扩展变量，`amount` 字段用于多级审批金额网关 |

### Step 4：编写回调控制器

审批完成后，workflow 服务会回调业务模块。需要：

**4a. 在 Service 中实现回调处理方法：**

```java
@Override
@Transactional(rollbackFor = Exception.class)
public void handleApprovalCallback(Long id, int result, String comment) {
    YourEntity entity = getById(id);
    if (entity.getStatus() != Status.APPROVING.getCode()) {
        throw new BizException("当前状态不在审批中，无法处理回调");
    }

    // result: 2=通过, 3=驳回
    int newStatus = (result == 2)
            ? Status.APPROVED.getCode()
            : Status.REJECTED.getCode();

    update(new LambdaUpdateWrapper<YourEntity>()
            .eq(YourEntity::getId, id)
            .set(YourEntity::getStatus, newStatus));

    // 通过后的业务逻辑（如更新关联数据、生成下游单据等）
    if (result == 2) {
        doAfterApproved(entity);
    }
}
```

**4b. 编写回调控制器（每个业务模块一个）：**

```java
package com.asset.yourmodule.callback;

import com.asset.api.workflow.dto.ApprovalCallbackDTO;
import com.asset.common.exception.BizException;
import com.asset.common.model.R;
import lombok.RequiredArgsConstructor;
import lombok.extern.slf4j.Slf4j;
import org.springframework.web.bind.annotation.*;

@Slf4j
@RestController
@RequestMapping("/yourprefix/internal")
@RequiredArgsConstructor
public class YourApprovalCallbackController {

    private final YourBusinessService yourBusinessService;
    // 注入更多需要审批的 Service...

    @PostMapping("/approval-callback")
    public R<Void> onCallback(@RequestBody ApprovalCallbackDTO dto) {
        log.info("[回调] type={}, bizId={}, result={}",
                dto.getBusinessType(), dto.getBusinessId(), dto.getResult());

        switch (dto.getBusinessType()) {
            case "YOUR_BUSINESS_TYPE" -> yourBusinessService.handleApprovalCallback(
                    dto.getBusinessId(), dto.getResult(), dto.getComment());
            // 添加更多 case...
            default -> throw new BizException("未知业务类型: " + dto.getBusinessType());
        }
        return R.ok();
    }
}
```

**ApprovalCallbackDTO 字段说明：**

| 字段 | 类型 | 说明 |
|------|------|------|
| processInstanceId | String | 流程实例 ID |
| businessType | String | 业务类型枚举值 |
| businessId | Long | 业务单据 ID |
| result | Integer | **2=通过，3=驳回** |
| comment | String | 最终审批意见 |
| approverId | Long | 审批人 ID |
| approverName | String | 审批人姓名 |

**现有回调控制器参考：**

| 模块 | 类名 | 路径 | 处理的业务类型 |
|------|------|------|---------------|
| asset-investment | InvApprovalCallbackController | `/inv/internal/approval-callback` | INV_INTENTION / INV_OPENING / INV_RENT_DECOMP |
| asset-operation | OprApprovalCallbackController | `/opr/internal/approval-callback` | OPR_CONTRACT_CHANGE / OPR_TERMINATION |
| asset-finance | FinApprovalCallbackController | `/fin/internal/approval-callback` | FIN_WRITE_OFF / FIN_DEDUCTION / FIN_ADJUSTMENT |

## 三、前端集成（3 步）

### Step 1：业务页面添加审批组件

系统提供两个公共审批组件，位于 `src/components/workflow/`：

**审批时间线（展示审批进度）：**

```vue
<template>
  <!-- 在业务详情页中添加审批流程展示 -->
  <el-divider content-position="left">审批流程</el-divider>
  <ApprovalTimeline :business-type="'INV_INTENTION'" :business-id="formData.id" />
</template>

<script setup lang="ts">
import ApprovalTimeline from '@/components/workflow/ApprovalTimeline.vue'
</script>
```

`ApprovalTimeline` 组件支持三种数据源（任选其一）：

| Props | 说明 |
|-------|------|
| `businessType` + `businessId` | 按业务单据自动查询审批记录 |
| `processInstanceId` | 按流程实例 ID 查询 |
| `records` | 手动传入审批记录数组 |

**审批发起弹窗（提交审批时使用）：**

```vue
<template>
  <el-button type="primary" @click="approvalDialogVisible = true">提交审批</el-button>

  <ApprovalDialog
    v-model:visible="approvalDialogVisible"
    title="提交XXX审批"
    :loading="submitting"
    @confirm="handleSubmitApproval"
  />
</template>

<script setup lang="ts">
import { ref } from 'vue'
import ApprovalDialog from '@/components/workflow/ApprovalDialog.vue'

const approvalDialogVisible = ref(false)
const submitting = ref(false)

async function handleSubmitApproval() {
  submitting.value = true
  try {
    await yourApi.submitApproval(currentId.value)
    ElMessage.success('已提交审批')
    approvalDialogVisible.value = false
  } finally {
    submitting.value = false
  }
}
</script>
```

## 四、新增业务类型（完整 Checklist）

当需要为新功能添加审批流程时，按以下清单逐项完成：

### 后端

- [ ] **枚举注册**：在 `ApprovalBusinessType` 中新增枚举值
  ```java
  // asset-api-workflow/.../enums/ApprovalBusinessType.java
  NEW_TYPE("新业务审批", "asset-yourmodule"),
  ```

- [ ] **Service 提交审批**：在业务 ServiceImpl 中注入 `ApprovalService`，实现 `submitApproval()` 方法

- [ ] **Service 处理回调**：实现 `handleApprovalCallback(Long id, int result, String comment)` 方法

- [ ] **回调控制器**：在模块的 `XxxApprovalCallbackController` 中添加 `case "NEW_TYPE"` 分支
  - 若模块已有回调控制器，直接添加 case
  - 若为全新模块，参照 Step 4b 创建回调控制器

- [ ] **流程定义种子数据**：在 `sql/workflow_init.sql` 中添加 INSERT 记录
  ```sql
  INSERT INTO wf_process_definition (process_key, process_name, business_type, ...)
  VALUES ('NEW_TYPE', '新业务审批', 'NEW_TYPE', ...);
  ```

- [ ] **BPMN 流程文件**（Flowable 模式需要）：
  在 `asset-workflow/src/main/resources/processes/` 下创建 `new_type_approval.bpmn20.xml`

### 前端

- [ ] **业务类型选项**：在 `src/api/workflow/task.ts` 的 `BUSINESS_TYPE_OPTIONS` 中添加
  ```ts
  { value: 'NEW_TYPE', label: '新业务审批' },
  ```

- [ ] **路由映射**：在 `BUSINESS_ROUTE_MAP` 中添加跳转路径

- [ ] **业务页面**：添加 `ApprovalTimeline` 和 `ApprovalDialog` 组件

## 五、Mock 模式与 Flowable 模式

### Mock 模式（默认）

```yaml
approval:
  engine: mock   # 或不配置
```

**行为：**
1. `approvalService.submit()` 立即返回 `MOCK-{type}-{id}-{timestamp}` 格式的 ID
2. **不启动真实流程**，不调用 asset-workflow 服务
3. 业务模块需自行决定何时触发回调（Mock 模式下审批自动通过）

**适用场景：** 本地开发、单元测试、不需要审批的功能调试

### Flowable 模式

```yaml
approval:
  engine: flowable
```

**行为：**
1. `approvalService.submit()` 通过 Feign 调用 asset-workflow（端口 8010）
2. Flowable 引擎启动 BPMN 流程，根据金额网关分配审批节点
3. 审批人在前端审批中心（`/workflow/todo`）处理待办
4. 审批完成后 asset-workflow 通过 Feign 回调业务模块

**前置条件：**
- asset-workflow 服务已启动（端口 8010）
- `sql/workflow_init.sql` 已执行（流程定义种子数据）
- Flowable 首次启动会自动建表（约 40 张 ACT_* / FLW_* 表，需等待约 30 秒）

**切换方式：** 修改 `application-dev.yml` 中的 `approval.engine` 值后重启服务即可，**业务代码无需任何修改**。

## 六、多级审批流程说明

所有 BPMN 流程采用统一的三级金额网关模式：

```
                                     是                              是
发起 → 部门领导审批 → [金额>=10万?] ───→ 副总裁审批 → [金额>=50万?] ───→ 总经理审批 → 结束
                         │ 否                            │ 否
                         ▼                               ▼
                       结束                             结束
```

| 审批级别 | 条件 | 审批人分配方式 |
|---------|------|--------------|
| 部门领导 | 始终执行 | 流程变量 `${deptLeaderId}` 指定 |
| 副总裁 | `amount >= 100000` | 候选组 `ROLE_VP` |
| 总经理 | `amount >= 500000` | 候选组 `ROLE_GM` |

**触发多级审批：** 在提交时传入 `variables.amount`：

```java
ApprovalSubmitDTO dto = new ApprovalSubmitDTO();
dto.setBusinessType("YOUR_TYPE");
dto.setBusinessId(id);
dto.setTitle("审批标题");
dto.setVariables(Map.of("amount", entity.getTotalAmount()));  // 金额决定审批层级
```

- 金额 < 10万 → 仅部门领导审批
- 10万 ≤ 金额 < 50万 → 部门领导 + 副总裁
- 金额 ≥ 50万 → 部门领导 + 副总裁 + 总经理

## 八、常见问题（FAQ）

### Q1：Mock 模式下提交审批后单据一直处于"审批中"怎么办？

Mock 模式下 `submit()` 只返回 Mock ID，不会自动触发回调。业务模块需要在 Mock 模式下自行处理状态流转，或在控制器中手动调用 `handleApprovalCallback(id, 2, "Mock自动通过")`。

### Q2：切换到 Flowable 模式后启动报错 `Connection refused`？

确保 asset-workflow（端口 8010）已启动，且 Nacos 注册中心可用。Flowable 模式通过 Feign 调用 workflow 服务，服务未注册时会连接失败。

### Q3：Flowable 首次启动非常慢？

正常现象。Flowable 首次启动需要自动创建约 40 张引擎表（`ACT_*`、`FLW_*`），耗时约 30 秒。后续启动无此问题。

### Q4：如何在不启动 asset-workflow 的情况下开发业务功能？

保持 `approval.engine: mock`（默认值）即可。Mock 模式不依赖任何外部服务。

### Q5：审批回调的 result 值含义？

| result | 含义 | 业务模块应做的操作 |
|--------|------|------------------|
| 2 | 通过 | 将单据状态更新为"已通过"，执行后续业务逻辑 |
| 3 | 驳回 | 将单据状态更新为"已驳回"，允许修改后重新提交 |

### Q6：财务模块回调为什么不直接用 businessId？

财务模块（核销/减免/调整）使用 `processInstanceId`（即 approvalId）而非 businessId 来查找单据，因为财务单据通过 `approvalId` 字段关联流程。`FinApprovalCallbackController` 中的 `dto.getProcessInstanceId()` 对应的就是 `submit()` 返回的审批 ID。

### Q7：如何自定义审批流程（不使用默认的三级金额网关）？

1. 在 `asset-workflow/src/main/resources/processes/` 下创建自定义 BPMN 文件
2. 在 `wf_process_definition` 表中配置该流程定义
3. BPMN 中的 `process id` 必须与 `process_key` 一致
4. 可在前端流程定义管理页面（`/workflow/definitions`）直接编辑 BPMN XML

### Q8：前端审批中心在哪里？

| 页面 | 路由 | 功能 |
